#### 项目复盘
##### 开发过程中遇到的一些问题
- 环境导致或者运维导致的问题
1. .env文件，在linux 系统下，引号解析有问题，不能用引号包裹，否则会导致报错，在Windows系统下，引号解析没有问题
2. docker-compose文件中，app应用和DB、Redis、Minio这些都处于bridge网络的时候，app应用需要连接的HOST，要使用app应用的名称，否则会导致报错
3. https 链接下，不能直接下载http协议的文件，否则会导致报错
4. 在阿里云这样的Linux服务器下，docker 需要配置dns, 否则无法联网，无法下载镜像文件
5. 关于docker 配置和使用的问题。配置端口映射的时候，不绑定IP地址更方便一点，因为这样可以在任何地方访问
```yaml
services:
    ports:
      - '5672:5672'
      - '15672:15672'
```

- 代码逻辑导致的问题
1. rabbitMQ Email Consumer 首次注册失败，又没有重试机制，会导致邮件发送失败。就是在onModuleInit()中，会尝试注册一次，如果失败，会记录错误，但是不会阻止应用继续启动。
如果注册成功，就会监听队列，进行消费。修改的方法，是添加重试机制。
```ts
try {
  await this.mailQueueService.setupTopology();
  await this.rabbitmqService.consume(queue, handler);
} catch (error) {
  this.logger.error({ event: "rabbitmq.consumer_start_failed", err: error });
}
```
```ts
  private scheduleStartRetry() {
    if (this.stopped || this.started || this.startTimer) return;
    this.startTimer = setTimeout(() => {
      this.startTimer = undefined;
      void this.start();
    }, 5000);
    this.startTimer.unref();
  }
```
2. Nestjs 中动态参数路由放在最后，避免拦截 friends/groups 等静态路由。我之前就是因为这个原因，导致了路由拦截问题，一致无法正常调用接口，就是接口出现的先后顺序，也就代表了路由的注册顺序，模块的注册顺序，也决定着导致路由的注册顺序。先匹配的handler，就会先执行，后面的不再参与匹配
3. 当前 MinIO 的 public 桶未开放匿名读取，因此不能使用无签名的固定对象 URL直接访问。项目通过 presignedGetObject() 为私有对象生成一小时有效的预签名 URL，浏览器可以直接访问该 URL。response-content-disposition: inline 用于提示浏览器以内联方式预览对象。
如果需要永久公开访问，可为桶或指定对象前缀配置只读匿名策略，但需评估文件泄露风险。例如 mc anonymous set download
```ts
const url = await this.generatePresignedUrl(name, "minio.preview_url", () =>
      this.minioPublicClient.presignedGetObject("public", name, 3600, {
        // 这一段必须
        "response-content-disposition": "inline",
      }),
    );
```
