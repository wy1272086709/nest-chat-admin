# 会话列表 N+1 查询优化复盘

## 问题背景

`GET /api/chat/rooms` 会返回当前用户的会话列表，每个会话还需要包含：

- 房间及成员信息。
- 当前用户的清空时间。
- 清空之后的最后一条消息。
- 上次已读或清空之后的未读消息数。

原实现先查询全部会话，然后在 `memberships.map()` 中为每个会话分别查询清空状态、最后消息和未读数。

## 为什么是 N+1

假设用户有 `N` 个会话，原实现的业务查询数为：

```text
1 次：查询全部会话
N 次：查询每个会话的清空状态
N 次：查询每个会话的最后消息
N 次：统计每个会话的未读数

总计：1 + 3N
```

`Promise.all()` 只能让这些查询并发执行，不能减少查询数量。并发查询还可能瞬间占满数据库连接池。数据库位于远程网络时，每次往返延迟会被 `3N` 进一步放大。

## 修复目标

- 查询数量不能随会话数线性增长。
- 保持接口返回结构和原有业务语义不变。
- 不能把全部历史消息加载到 Node.js 后再统计。
- 所有 SQL 参数必须参数化，不能拼接用户输入。

## 修复方案

修复后使用固定 3 次业务查询：

```text
1. Prisma 查询全部会话、房间和成员
2. 参数化 SQL 批量计算所有会话的快照
3. Prisma 按最后消息 ID 批量加载消息和发送者
```

批量快照一次返回：

```ts
type ConversationSnapshotRow = {
  roomId: string;
  clearedAt: Date | null;
  lastMessageId: string | null;
  unreadCount: number;
};
```

最后在内存中通过 `Map` 按 `roomId` 和 `lastMessageId` 组装结果。Map 查找为常数时间，整体组装复杂度约为 `O(N)`，但不会产生额外数据库往返。

## 为什么使用参数化 SQL

Prisma `groupBy()` 可以按房间统计数量，但这里每个房间的未读阈值不同：

```text
max(lastReadAt, clearedAt)
```

最后消息还需要遵守每个房间不同的 `clearedAt`。如果强行只使用普通 Prisma 查询，通常需要：

- 继续逐房间查询；或者
- 拉取大量历史消息到应用层过滤。

因此这里使用 `Prisma.sql` 和 `$queryRaw` 做参数化批量计算。`${userId}` 会作为绑定参数传给 PostgreSQL，不是字符串拼接。

## 关键业务语义

最后消息：

```text
未删除
并且 createdAt > clearedAt
按 createdAt 倒序取第一条
```

未读消息：

```text
不是当前用户发送
未删除
并且 createdAt > max(lastReadAt, clearedAt)
```

当 `lastReadAt` 或 `clearedAt` 为空时，SQL 使用 PostgreSQL 的 `-infinity` 时间作为下界，保持原实现“没有阈值就统计全部”的语义。

## 索引配套

原表只有：

```prisma
@@unique([roomId, userId])
```

它适合通过房间和用户查单条成员记录，但不适合从 `userId` 开始查询用户全部活跃会话。因此新增：

```prisma
@@index([userId, status])
```

已有的消息索引：

```prisma
@@index([roomId, createdAt])
```

可支持按房间和时间定位最后消息。迁移文件为：

```text
20260714000000_add_room_member_lookup_index
```

## 修复前后压测

测试接口：`GET /api/chat/rooms`，每档持续 10 秒，使用相同本机服务和远程 PostgreSQL。

| 并发 | 修复前 QPS | 修复后 QPS | 修复前平均延迟 | 修复后平均延迟 | 修复前 P90 | 修复后 P90 |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 5 | 8.6 | 9.4 | 573ms | 514ms | 807ms | 638ms |
| 10 | 19.7 | 18.7 | 494ms | 522ms | 603ms | 681ms |
| 20 | 25.9 | 33.7 | 746ms | 504ms | 893ms | 576ms |

20 并发下：

- QPS 提升约 30%。
- 平均延迟下降约 32%。
- P90 下降约 36%。

低并发结果存在短时网络抖动，不能只凭单轮 10 秒测试断言稳定收益。修复后的 20 并发测试仍出现 2 次客户端超时，P99 仍接近 1 秒，说明远程数据库、鉴权阶段的用户查询和 Redis 会话检查、数据库连接池等仍可能贡献尾延迟。

本次优化最确定的收益不是某一个 QPS 数字，而是把数据库查询复杂度从 `O(N)` 次往返降为固定 3 次，使性能不再随会话数量快速恶化。

## 验证方式

已完成：

- TypeScript 编译。
- Prisma Client 重新生成。
- 数据库索引迁移应用。
- 修复后接口冒烟测试，全部返回 HTTP 200。
- 5、10、20 并发阶梯复测。

正式上线前还应补充：

- 空会话列表。
- 没有清空记录。
- 清空后没有新消息。
- `lastReadAt` 早于或晚于 `clearedAt`。
- 自己发送的消息不计入未读。
- 已删除消息不作为最后消息且不计入未读。
- 多个会话返回结构与优化前一致。

## 面试表达

可以按以下顺序回答：

> 我在会话列表压测中发现并发增加后吞吐增长变慢、延迟快速上升。检查代码发现虽然使用了 Promise.all，但每个会话仍分别查询清空状态、最后消息和未读数，查询数是 1+3N。Promise.all 只改变并发方式，不解决 N+1，还可能打满连接池。

> 我把每个会话需要的 clearedAt、lastMessageId 和 unreadCount 用一次参数化 SQL 批量计算，再批量查询最后消息详情，最终通过 Map 组装，将业务查询数固定为 3 次。同时为 RoomMember 增加 userId、status 复合索引。

> 修复后 20 并发 QPS 提升约 30%，平均延迟下降约 32%。更重要的是查询次数不再随会话数线性增长。测试仍存在远程数据库带来的 P99 尾延迟，因此后续会继续结合数据库慢查询、连接池和事件循环指标定位，而不会只看平均 QPS。
