# API 压测报告与基准

## 测试目标

本次测试用于建立当前项目的初始性能基线，不等同于生产容量认证。

- 日期：2026-07-14
- 接口：`GET /api/chat/rooms`
- 工具：Autocannon
- 类型：只读接口阶梯压测
- 服务：本机 `127.0.0.1:3000`
- 外部依赖：远程 PostgreSQL、Redis
- 鉴权：复用已有活跃会话的 `jti`，生成 5 分钟临时 JWT
- 安全边界：不创建用户、不发送邮件、不投递 RabbitMQ、不修改登录会话

## 结果判断

可靠性方面，本轮所有请求均返回 200，没有连接错误或超时，短时稳定性合格。

延迟方面，当前结果不理想。10 并发下 P90 为 603ms、P99 为 842ms；20 并发下 P99 超过 1 秒。对于会话列表这类高频只读接口，用户会感受到刷新延迟。

容量方面，10 并发增加到 20 并发时：

- QPS 从 19.7 增加到 25.9，仅增加约 31%。
- 平均延迟从 494ms 增加到 746ms，增加约 51%。

吞吐增长开始放缓，同时延迟明显升高，说明当前链路在 10 到 20 并发之间已出现饱和迹象。当前结果不建议直接解释为“系统只能支持 25 QPS”，因为测试时间较短，数据库为远程依赖，且没有同步采集 CPU、事件循环和数据库指标。

## 主要瓶颈推断

`listConversations()` 存在随会话数量增长的 N+1 查询：

```text
1 次查询会话成员关系
+ 每个会话 1 次查询清空状态
+ 每个会话 1 次查询最后消息
+ 每个会话 1 次统计未读数
```

除此之外，每个请求的 JWT 校验还会查询用户并访问 Redis 检查当前 `jti`。远程 PostgreSQL 的网络往返延迟会被多次查询放大。因此优先检查 SQL 次数和数据库耗时，而不是先怀疑 Pino 或 NestJS 框架性能。

建议的优化顺序：

1. 为一次请求记录 Prisma 查询数量和总数据库耗时。
2. 将清空状态、最后消息和未读数改为批量查询或聚合查询。
3. 检查 `RoomMember(userId, status)`、`Message(roomId, createdAt)` 等查询索引。
4. 在与应用同区域的 PostgreSQL 上重新测试，排除公网网络延迟。
5. 使用 production 构建运行，并避免开发环境的 pretty 日志影响测试。

## N+1 修复复测

`listConversations()` 已将 `1 + 3N` 次业务查询优化为固定 3 次查询，并新增 `RoomMember(userId, status)` 复合索引。

| 并发连接 | 修复后平均 QPS | 修复后平均延迟 | 修复后 P90 | 修复后 P99 | 错误/超时/非 2xx |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 5 | 9.4 | 514ms | 638ms | 1178ms | 0 / 0 / 0 |
| 10 | 18.7 | 522ms | 681ms | 991ms | 0 / 0 / 0 |
| 20 | 33.7 | 504ms | 576ms | 972ms | 2 / 2 / 0 |

20 并发时 QPS 较修复前提升约 30%，平均延迟下降约 32%，P90 下降约 36%。短时测试仍有网络抖动和 2 次超时，P99 仍需继续优化。详细修复分析见 `docs/chat-list-conversations-n-plus-one-retrospective.md`。

## 什么结果比较合理

不存在适用于所有项目的统一 QPS，合理标准应以接口 SLO 和预计峰值流量为准。当前个人项目可以先采用以下工程目标：

| 指标 | 建议目标 |
| --- | --- |
| HTTP 错误率 | 小于 0.1% |
| 压测超时 | 0 |
| 高频只读接口 P50 | 小于 100ms |
| 高频只读接口 P95 | 小于 300ms |
| 高频只读接口 P99 | 小于 500ms |
| 普通写接口 P95 | 小于 500ms |
| CPU 持续利用率 | 小于 70% |
| 内存 | 压测期间趋于稳定，无持续增长 |
| Node.js 事件循环延迟 P99 | 小于 100ms |
| 数据库连接池 | 无长期等待、无连接耗尽 |

对于依赖第三方邮件、对象存储等外部服务的接口，应单独定义更宽松的 SLO，不能和纯数据库读接口使用同一标准。

## 重点观察指标

### 1. 延迟分位数

平均值容易掩盖慢请求，优先看 P95 和 P99。P99 表示 99% 的请求都不超过该耗时。

### 2. 错误率和超时

重点检查非 2xx、连接错误、超时，以及业务响应中 HTTP 200 但 `result=false` 的情况。后者仅靠压测工具的状态码统计无法识别，需要增加响应断言。

### 3. 吞吐与延迟拐点

逐步增加并发。当 QPS 不再近似增长，但 P95/P99 快速上升时，说明系统开始饱和。稳定容量应选择在拐点之前，并保留至少 30% 余量。

### 4. 资源和依赖指标

压测时同步观察：

- 应用 CPU、RSS 内存、事件循环延迟和 GC。
- PostgreSQL 慢查询、活跃连接、锁等待和连接池等待。
- Redis 命令延迟和连接数。
- RabbitMQ 的 Ready、Unacked、Publish Rate、Ack Rate 和 DLQ。
- 日志写入量，避免同步输出或日志采集成为瓶颈。

## 如何复测

项目提供只读压测脚本：

```bash
pnpm load:test:readonly
```

默认参数：

```text
URL=http://127.0.0.1:3000/api/chat/rooms
并发连接=20
持续时间=15 秒
```

自定义参数：

```bash
LOAD_TEST_CONNECTIONS=10 \
LOAD_TEST_DURATION=30 \
LOAD_TEST_URL=http://127.0.0.1:3000/api/chat/rooms \
pnpm load:test:readonly
```

正式基准建议先预热 30 秒，再分别运行 10、20、50 并发，每档至少持续 2 到 5 分钟。每档重复 3 次，取中位数，并同时采集应用与数据库指标。

不要直接压测 `sendEmail`、注册、修改状态等有副作用接口。此类场景应使用独立测试环境、专用数据，并 Mock SMTP 等外部服务。
